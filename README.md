# Mini NPU Simulator

## 1. 프로젝트 개요

이 프로젝트는 입력 패턴과 필터의 같은 위치 원소를 곱한 뒤 모두 더하는 MAC(Multiply-Accumulate) 연산을 이용해, 입력 패턴이 `Cross`에 더 가까운지 `X`에 더 가까운지 판별하는 콘솔 기반 프로그램이다.

프로그램은 두 가지 실행 모드를 제공한다.

- **모드 1**: 사용자가 3×3 필터 2개와 3×3 패턴 1개를 직접 입력해 점수와 판정 결과를 확인한다.
- **모드 2**: `data.json`의 `filters`, `patterns`를 읽어 전체 케이스를 자동 분석하고 PASS/FAIL 결과를 요약한다.

이 구현의 핵심 목표는 다음과 같다.

- MAC 연산을 반복문으로 직접 구현한다.
- `expected('+', 'x')`, `filter key('cross', 'x')`를 내부 표준 라벨 `Cross`, `X`로 정규화한다.
- 부동소수점 비교 오차를 고려해 `epsilon` 정책을 적용한다.
- 결과뿐 아니라 실패 사유와 복잡도까지 설명 가능하게 만든다.

## 5. 핵심 구현 내용

### 5-1. MAC 연산

이 프로그램의 핵심 연산은 MAC(Multiply-Accumulate)이다.

입력 패턴 `pattern`과 필터 `filter`에 대해 같은 위치의 값을 곱한 뒤 모두 더해 점수를 계산한다.

```text
score = Σ(pattern[i][j] * filter[i][j])
```

점수가 높을수록 해당 필터와 입력 패턴이 더 유사하다고 판단한다.

### 5-2. 라벨 정규화

데이터셋에는 동일한 의미의 라벨이 다른 문자열로 표현될 수 있다.
예를 들어 `+`, `cross`, `x`가 서로 다른 위치에서 사용될 수 있다.

이를 그대로 비교하면 문자열 불일치로 잘못된 FAIL이 발생할 수 있으므로, 프로그램 내부에서는 아래 기준으로 라벨을 정규화했다.

* `+` → `Cross`
* `cross` → `Cross`
* `x` → `X`

이 정규화는 `normalize_label()` 함수에서 수행한다.

### 5-3. 크기 추출 및 필터 매칭

`patterns`의 각 키는 `size_{N}_{idx}` 형식을 가진다.
예를 들어 `size_13_2`라면, 이 케이스는 `13×13` 패턴이라는 뜻이다.

프로그램은 `extract_size_from_case_id()` 함수로 N을 추출한 뒤, `filters`의 `size_N`에 해당하는 Cross/X 필터를 선택한다.

즉, 패턴과 필터의 연결은 아래 기준으로 이뤄진다.

* `size_3_*` → 3×3 필터 사용
* `size_5_*` → 5×5 필터 사용
* `size_13_*` → 13×13 필터 사용
* `size_25_*` → 25×25 필터 사용

### 5-4. 행렬 검증

잘못된 입력이나 잘못된 데이터셋 구조가 들어와도 프로그램이 바로 종료되지 않도록, `validate_matrix()` 함수로 다음 항목을 검증했다.

* 리스트 형식인지
* 비어 있지 않은지
* 2차원 정사각 행렬인지
* 모든 원소가 숫자인지
* 기대 크기와 실제 크기가 일치하는지

검증 실패 시에는 예외로 종료하지 않고, 실패 사유를 문자열로 반환하도록 설계했다.

### 5-5. epsilon 비교 정책

부동소수점 연산에서는 미세한 오차가 발생할 수 있다.
예를 들어 수학적으로 같은 값이어도 메모리 내부 표현에서는 아주 작은 차이가 생길 수 있다.

이를 고려해 두 점수를 비교할 때 단순히 `==`를 사용하지 않고, 아래 기준을 사용했다.

```text
abs(score_cross - score_x) < 1e-9
```

이 조건을 만족하면 `UNDECIDED`로 처리한다.

## 6. 코드 구조 및 설계 설명

### 6-1. MAC 연산 흐름

MAC 점수 계산의 전체 흐름은 다음과 같다.

1. 입력 패턴과 필터를 준비한다.
2. 행렬의 각 위치 `(i, j)`를 순회한다.
3. `pattern[i][j] * filter[i][j]`를 계산한다.
4. 계산 결과를 누적 변수 `total`에 더한다.
5. 모든 위치 순회가 끝나면 최종 점수를 반환한다.

즉, 구조적으로는 **입력 → 위치별 곱셈 → 누적 합산 → 점수 반환** 흐름으로 구현했다.

### 6-2. `patterns`와 `filters` 매칭 설계

패턴 데이터는 `size_{N}_{idx}` 형태의 키를 가진다.
이 키에서 N을 추출하면 해당 패턴이 어떤 크기인지 알 수 있다.

필터 데이터는 `filters["size_N"]`처럼 크기 기준으로 저장되어 있으므로, 프로그램은 다음 과정을 거쳐 매칭한다.

1. `case_id`에서 N 추출
2. `filters`를 순회하면서 `size_N`을 정수 크기로 변환
3. `filter_map[size] = {"Cross": ..., "X": ...}` 형태로 저장
4. 각 패턴 케이스 분석 시 `filter_map[N]`을 참조

이 방식의 장점은 다음과 같다.

* 크기 기준 접근이 직관적이다.
* 필터를 한 번만 정리하면 여러 케이스에서 재사용할 수 있다.
* 잘못된 크기나 누락된 필터를 별도 실패 사유로 처리하기 쉽다.

### 6-3. `normalize_label()`를 분리한 이유

라벨 정규화를 별도 함수로 분리한 이유는 비교 기준을 한곳에서 통일하기 위해서다.

장점은 다음과 같다.

* 라벨 규칙이 바뀌어도 한 함수만 수정하면 된다.
* `expected`, `filter key` 모두 같은 규칙으로 처리할 수 있다.
* 코드 중복이 줄어든다.
* 비교 실수 가능성이 줄어든다.

### 6-4. 시간 측정 경계

성능 측정에서는 입출력 시간을 제외하고, 실제 연산 비용만 측정하는 것이 중요하다.

이 프로그램은 `measure_average_ms()` 함수에서 아래 경계를 기준으로 측정한다.

* 시작: `mac_score(pattern, filt)` 호출 직전
* 종료: `mac_score(pattern, filt)` 호출 직후

즉, 콘솔 입력/출력, JSON 파싱, 결과 출력 시간은 제외하고 **연산 구간만** 측정한다.

### 7-2. 실패 케이스 목록

* `[case_id]`: `[실패 사유]`
* `[case_id]`: `[실패 사유]`
* `[case_id]`: `[실패 사유]`

### 7-3. 실패 원인 분석 및 복잡도 분석

1. 실패 케이스는 크게 데이터 형식 오류, 라벨 정규화 실패, 예상 라벨 불일치, 동점 처리로 나눌 수 있다.
2. `case_id` 형식이 잘못되면 패턴 크기를 추출할 수 없어서 필터를 선택할 수 없다.
3. 행렬 크기가 기대한 N과 다르면 MAC 연산 전 검증 단계에서 FAIL로 처리된다.
4. 숫자가 아닌 값이 포함되면 연산 자체가 불가능하므로 검증 실패로 처리된다.
5. `expected`가 `+`, `cross`, `x` 외의 값이면 내부 표준 라벨로 변환할 수 없어 FAIL이 된다.
6. Cross 점수와 X 점수의 차이가 `epsilon`보다 작으면 `UNDECIDED`로 처리된다.
7. 모드 2에서는 `UNDECIDED`가 expected와 일치하지 않으므로 FAIL로 집계된다.
8. MAC 연산은 `n × n` 행렬의 모든 칸을 한 번씩 방문하므로 연산 횟수는 `N²`이다.
9. 따라서 입력 크기가 커질수록 실행 시간도 `O(N²)` 경향을 보인다.
10. 측정 결과가 증가하는 이유는 필터와 패턴의 각 위치에 대해 곱셈과 덧셈을 반복 수행하기 때문이다.
11. 이 프로그램은 대규모 병렬 계산 대신 CPU의 직렬 반복문으로 MAC을 수행하므로, 크기가 커질수록 지연이 더 크게 느껴질 수 있다.
12. 즉, 실패 원인은 데이터 품질과 비교 정책에 주로 영향을 받고, 실행 시간은 행렬 크기 증가에 가장 큰 영향을 받는다.

### 7-4. 성능 측정 결과

| 크기    |   평균 시간(ms) | 연산 횟수(N²) |
| ----- | ----------: | --------: |
| 3×3   | `[실행 후 입력]` |         9 |
| 5×5   | `[실행 후 입력]` |        25 |
| 13×13 | `[실행 후 입력]` |       169 |
| 25×25 | `[실행 후 입력]` |       625 |

성능 해석:

* 행렬 크기가 커질수록 연산 횟수가 증가한다.
* 연산 횟수 증가는 MAC 반복 횟수 증가와 직접 연결된다.
* 따라서 평균 실행 시간도 전체적으로 증가하는 경향을 보인다.
* 이 결과는 MAC 구현이 모든 원소를 순회하는 `O(N²)` 구조라는 점과 일치한다.

## 9. 핵심 기술 원리 설명

### 9-1. MAC 점수가 높을수록 더 유사하다고 볼 수 있는 이유

MAC는 같은 위치의 값을 곱해 모두 더한다.
즉, 패턴과 필터가 비슷한 위치에서 같은 형태를 많이 가지면 곱셈 결과가 더 많이 누적된다.

따라서 특정 필터와 유사한 패턴일수록 해당 필터의 점수가 더 높아진다.

### 9-2. 왜 O(N²)인가

입력 크기가 N이면 행이 N개, 열이 N개이므로 총 원소 수는 `N × N`개다.
MAC 연산은 이 모든 원소를 한 번씩 방문해 곱셈과 덧셈을 수행한다.

그래서 전체 시간 복잡도는 `O(N²)`이다.

### 9-3. 부동소수점 오차와 epsilon이 필요한 이유

컴퓨터는 실수를 2진수 기반으로 저장하기 때문에, 일부 값은 정확히 표현되지 않는다.
이 때문에 계산 결과가 사람이 기대하는 값과 아주 조금 다를 수 있다.

따라서 두 점수가 사실상 같다고 봐야 하는 상황에서도 단순 비교를 하면 다른 값처럼 보일 수 있다.
이 문제를 줄이기 위해 epsilon 기준을 사용했다.

### 9-4. mode1 vs mode2 동점 처리 규칙 설명

요구사항 관점에서는 mode1은 사용자에게 직관적인 결과를 보여주는 모드이고, mode2는 데이터셋 기준 PASS/FAIL을 평가하는 모드다.

* mode1에서는 사용자 관점에서 “판정 불가”처럼 이해하기 쉬운 표현이 적합하다.
* mode2에서는 평가 로직 기준으로 `UNDECIDED`를 명시하고 FAIL로 집계하는 방식이 적합하다.

현재 구현은 두 모드 모두 같은 `decide_label()` 로직을 사용하고 있으며, mode2에서는 `UNDECIDED`가 실제로 FAIL 집계에 반영된다.
제출 전 평가 기준이 더 엄격하다면 mode1 출력 문자열을 `판정 불가`로 분리하는 방식으로 보완할 수 있다.

## 10. 심층 인터뷰 대비

### 10-1. 새로운 라벨 `o`가 추가되면 어떻게 확장할 것인가

확장 순서는 다음과 같다.

1. `normalize_label()`에 `o` 규칙을 추가한다.
2. 내부 표준 라벨 체계를 `Cross`, `X`, `O`처럼 확장한다.
3. 필요한 경우 `decide_label()` 또는 다중 점수 비교 로직으로 확장한다.
4. 출력 형식을 새 라벨에 맞게 수정한다.
5. JSON 필터 로드 시 `O` 필터도 함께 읽도록 수정한다.
6. README와 테스트 케이스를 새 규칙에 맞게 갱신한다.

### 10-2. 1000×1000 패턴을 처리하면 어떤 문제가 생기는가

1000×1000이면 원소 수가 1,000,000개이므로 한 번의 MAC 연산도 매우 커진다.
이 경우 다음 문제가 발생할 수 있다.

* 반복문 실행 시간이 길어진다.
* 패턴과 필터를 메모리에 저장하는 비용이 커진다.
* 여러 케이스를 반복 처리하면 전체 수행 시간이 급격히 증가한다.

우선 적용할 최적화는 다음과 같다.

* 불필요한 출력과 I/O 줄이기
* 동일 크기 필터 재사용
* 가능하면 벡터화 또는 병렬 처리 구조 도입
* 측정 횟수 조정 및 큰 입력 전용 테스트 전략 분리

### 10-3. CPU 직렬 처리와 NPU 병렬 처리 관점의 병목

현재 구현은 파이썬 반복문으로 각 원소를 순차적으로 계산한다.
즉, CPU의 직렬 처리 방식에 가까운 구조다.

대규모 MAC에서는 병목이 다음 지점에 생긴다.

* 각 원소를 한 번씩 방문하는 반복문
* 곱셈과 덧셈의 누적 반복
* 큰 행렬을 여러 번 처리하는 전체 루프

반면 NPU는 다수의 MAC 연산을 병렬로 처리하는 데 최적화되어 있다.
따라서 문제 크기가 커질수록 CPU 기반 직렬 구현과 NPU 기반 병렬 구현의 성능 차이가 더 크게 나타난다.
